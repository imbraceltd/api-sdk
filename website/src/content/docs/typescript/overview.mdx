The `@imbrace/sdk` package is the official TypeScript/JavaScript client for the Imbrace Gateway. It provides fully-typed access to all Imbrace services for both server-side (Node.js) and client-side (browser) contexts.

### Key Features

| Feature                  | Detail                                                                               |
| ------------------------ | ------------------------------------------------------------------------------------ |
| **Full Type Safety**     | 100% TypeScript — every request and response is typed via shared `types/` interfaces |
| **Dual Authentication**  | API Key (server-to-server) or Access Token (client-side JWT / legacy token)          |
| **Auto Retry**           | 429 and 5xx errors retry up to 2 times with exponential backoff                      |
| **Request Cancellation** | AbortSignal propagation for in-flight request cancellation                           |
| **Multi-Environment**    | `develop` / `sandbox` / `stable` presets with per-service URL overrides              |
| **Streaming AI**         | AsyncGenerator-based streaming for AI completions                                    |

### How It Works

```
Your Code
   │
   ▼
ImbraceClient          ← single entry point, wires all resources
   ├── TokenManager    ← thread-safe token storage (OTP / JWT / API Key)
   └── HttpTransport   ← auth headers, retry, timeout, error mapping
         │
         ▼
   Imbrace Gateway     ← app-gatewayv2.imbrace.co (+ per-service URLs)
```

`ImbraceClient` creates one `HttpTransport` instance and shares it across all resource classes. Resources (`client.marketplace`, `client.platform`, etc.) are thin wrappers that construct the correct URL and call `transport.request()`.

### Internal Architecture

| Layer            | File                    | Responsibility                                          |
| ---------------- | ----------------------- | ------------------------------------------------------- |
| Entry Point      | `client.ts`             | Instantiates all resources, handles login helpers       |
| HTTP Transport   | `http.ts`               | Auth headers, retry logic, timeout, error mapping       |
| Token Manager    | `auth/token-manager.ts` | Stores and retrieves the current access token           |
| Service Registry | `service-registry.ts`   | Resolves per-service base URLs from environment presets |
| Resources        | `resources/*.ts`        | Domain-specific API method wrappers (29 resources)      |
| Types            | `types/*.ts`            | Shared TypeScript interfaces for all API shapes         |
| Errors           | `errors.ts`             | `AuthError`, `ApiError`, `NetworkError` definitions     |

### When to Use

| Scenario                                   | Recommended Auth                            |
| ------------------------------------------ | ------------------------------------------- |
| Node.js backend, cron jobs, data pipelines | `apiKey` → `x-api-key` header               |
| User-facing frontend after OTP/OAuth login | `accessToken` JWT → `Authorization: Bearer` |
| Legacy integration with opaque tokens      | `accessToken` opaque → `x-access-token`     |

:::tip
For most server-side use cases, use `apiKey`. For user-facing apps, authenticate with OTP and let the SDK manage the token automatically.
:::
